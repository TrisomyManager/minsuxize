## 1. 项目范围

### 1.1 首期覆盖的习俗主题范围

以下内容主题为首期可纳入记录的范围。这些范围用于界定记录主题，不等于实体清单，具体记录单位仍以「地方习俗实例」为核心。

| 主题类别 | 说明 | 示例 |
|---------|------|------|
| 岁时节令 | 与年节、节气相关的习俗实践 | 春节、清明、端午、中秋的地方过法 |
| 人生礼仪 | 出生、成年、婚嫁、丧葬等生命节点仪式 | 满月礼、婚礼流程、葬礼习俗 |
| 家族宗族 | 家族组织、祠堂活动、族谱相关习俗 | 祭祖、修谱、宗族聚会 |
| 信仰祭祀 | 民间信仰、神灵崇拜、祭祀活动 | 庙会、神诞、祈福仪式 |
| 生产习俗 | 与农、林、牧、渔、手工业相关的传统实践 | 开耕仪式、渔业禁忌 |
| 居住建筑 | 与房屋建造、入住相关的习俗 | 上梁、入宅、灶神祭祀 |
| 饮食供品 | 特定场合的饮食规范与供品配置 | 年夜饭构成、祭祀供品 |
| 口头传统 | 仪式语言、禁忌语、口彩、咒语等 | 婚礼吉祥话、丧葬避讳语 |
| 禁忌规矩 | 特定时间、地点、人群的行为约束 | 正月禁忌、丧事期间禁忌 |

### 1.2 实体类型范围（首期）

**首期核心实体（必须完成）：**
- 地点（Place）：从省到自然村的层级结构，作为所有记录的空间锚点
- 节日（Festival）：岁时节令的抽象词条
- 仪式（Ritual）：特定行为流程的抽象描述
- 地方习俗实例（Custom Instance）：特定地点的具体实践记录
- 来源资料（Source）：支持事实的证据材料

**二期扩展实体（首期不展开）：**
- 人群角色（ROL）、器物（OBJ）、信仰观念（BEL）、时间规则（TIM）、媒体资料（MED）

### 1.3 地域范围
- 首期仅覆盖 1 个试点区域
- 区域内精确到行政村、自然村级别
- 不追求首期覆盖全国

### 1.4 暂不纳入范围（首期）

| 类别 | 具体内容 | 暂不纳入原因 |
|-----|---------|------------|
| 非遗项目管理 | 非物质文化遗产的申报、评审、保护状态 | 属于行政管理范畴，与民俗记录目标不同 |
| 商业民俗活动 | 纯粹为旅游开发设计的表演性活动 | 非原生习俗，难以判定真实性 |
| 个人家族隐私 | 涉及具体个人姓名的家族内部事务 | 隐私保护风险，需额外伦理审查 |
| 争议性民俗 | 存在严重社会争议、可能引发伦理问题的习俗 | 需建立专门的伦理审查机制后再评估 |
| 跨境比较 | 中国境内少数民族与境外同族群的对比 | 范围过大，首期聚焦境内 |
| 古代复原 | 仅凭文献推测、无现存实践的习俗 | 难以验证，易陷入虚构 |

## 2. 内容边界

### 2.1 抽象词条与地方习俗实例必须分离

**规则**
- 抽象词条描述「是什么」，回答「春节是什么节日」
- 地方习俗实例描述「某地怎么做」，回答「福建省莆田市涵江区怎么过春节」
- 同一文件不得同时承担两种职责

**示例**
| 类型 | 正确写法 | 错误写法 |
|-----|---------|---------|
| 抽象词条 | 春节是中国农历新年，传统习俗包括拜年、吃年夜饭、放鞭炮等 | ❌ 福建省莆田市涵江区的春节是……（应拆分为单独的地方习俗实例） |
| 地方习俗实例 | 地点：福建省莆田市涵江区；相关节日：春节（FES-0001）；本地实践：正月初四过大年…… | ❌ 春节是一个重要节日，各地习俗不同，例如福建莆田……（混写） |

### 2.2 无来源时不能写成事实

**规则**
- 所有具体事实必须有可追溯的来源
- 来源类型包括：地方志、学术论文、田野调查记录、口述史访谈、历史影像等
- 缺乏来源时，只能记录为「待核查」或完全不记录，不得编造

**来源标注要求**
| 情况 | 处理方式 |
|-----|---------|
| 有明确来源 | 正常记录，绑定来源 ID |
| 来源存疑 | 记录为「待核查：据口口相传……」 |
| 完全无来源 | 不写入记录，或仅在讨论区标注 |

### 2.3 可记录差异版本，不强行统一

**规则**
- 同一习俗在不同地点、不同时间、不同人群中存在差异，这是正常状态
- 不得擅自判断「正宗」与「变体」
- 差异应作为并行的独立地方习俗实例记录，而非「错误纠正」

**示例**
- 实例 A：地点 X，春节吃饺子
- 实例 B：地点 Y，春节吃年糕
- 正确做法：分别记录为两个独立地方习俗实例，不标注孰正孰误
- 错误做法：以实例 A 为标准，说实例 B「不吃饺子是一种变异」

## 3. 记录原则

### 3.1 以地点和时间为硬骨架

**地点要求**
- 每条地方习俗实例必须绑定具体地点
- 地点精度优先到自然村级别，至少到行政村
- 地点使用标准化层级：国家 → 省 → 市 → 县/区 → 乡镇/街道 → 行政村 → 自然村 → 空间点位

**时间要求**
- 记录该实例观察或描述的时间范围
- 区分「发生时间」（习俗执行的时间点）与「记录时间」（资料产生的时间点）
- 传统周期性习俗需标明农历/公历对应关系

### 3.2 以来源证据为可信支撑

- 每个关键事实必须能追溯到具体来源
- 来源资料作为独立实体管理，可被多条地方习俗实例引用
- 来源需记录：类型、作者/机构、发布时间、获取方式、可信度评估

### 3.3 以地方习俗实例为核心节点

- 地方习俗实例是知识网络的基本单元
- 其他实体（地点、节日、仪式、来源）围绕地方习俗实例组织
- 查询和展示的默认入口是地方习俗实例，而非抽象词条

### 3.4 允许后续增补修订

- 记录以版本形式存在，不追求一次完备
- 新增实例不推翻旧实例，除非发现事实错误
- 修订需保留历史版本，注明修订原因

## 4. 结构原则

### 4.1 先结构，后扩充

- 必须先完成实体定义、字段规范、模板设计
- 结构稳定前，不盲目填充大量内容
- 结构变更需评估对已有数据的影响

### 4.2 先试点，后铺开

- 首期仅在一个试点区域验证完整流程
- 试点成功后再扩展到其他区域
- 每个新区域纳入前需评估现有结构是否适配

### 4.3 先核心实体，后扩展实体

**首期核心实体（必须完成）：**
- 地点、节日、仪式、地方习俗实例、来源资料

**二期扩展实体（首期不展开）：**
- 人群角色、器物、信仰观念、时间规则、媒体资料

## 5. 协作原则

### 5.1 多 AI 分模块

- 复杂任务分解为独立模块，分配给不同 AI 会话
- 每个模块有明确输入、输出、边界
- 模块间通过标准文档交接，不依赖口头说明

### 5.2 单轮只做一个模块

- 单次会话只处理一个模块文件
- 若模块含多个输出文件，需在开始前明确本次涉及文件
- 完成当前模块前，不跨模块扩写

### 5.3 输出必须可进入仓库

- 产出必须是可直接提交到 Git 仓库的 Markdown 文件
- 文件需包含完整的文档信息（模块编号、版本、修改记录）
- 输出内容不得包含「待补充」「稍后完善」等占位符

### 5.4 变更需记录

- 每次修改需在文件末尾更新修改记录表
- 重大变更需同步更新 `05_Operations/change_log.md`
- 引发审核分歧的变更需记录到 `05_Operations/review_log.md`

## 6. 禁止事项

以下行为在本项目中明确禁止：

| 序号 | 禁止事项 | 说明 |
|:---:|---------|------|
| 1 | 编造民俗事实 | 没有来源支撑时，不得虚构任何具体习俗内容 |
| 2 | 将抽象节日词条写成地方习俗实例 | 抽象词条只写通用概念，地方细节必须拆分为独立地方习俗实例 |
| 3 | 把评论性推测当事实写入 | 如「这可能源于古代祭祀」「据说有数百年的历史」等缺乏来源的推测 |
| 4 | 在未批准情况下重构整个目录 | 目录结构变更需经项目负责人或主架构 AI 批准 |
| 5 | 私自新增未定义字段 | 模板中不得使用字段表未定义的核心字段，如需新增应标记「待字段表确认」 |
| 6 | 混用多种命名风格 | 必须遵守《命名与版本规范》中的命名规则 |
| 7 | 删除或修改他人已发布的实体 ID | ID 一经分配不得复用或更改 |
| 8 | 在样板文档中使用虚构编号 | 样板中的未知编号应写为 `XXX-XXXX` 占位格式 |
