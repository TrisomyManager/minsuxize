# 中国民俗细则网站蓝图

## 1. 产品定位

### 1.1 项目使命

建立一个长期可积累、可审核、可追溯的中国民俗知识库，重点记录村落级别的地方性风俗，为传统文化传承提供尽可能具体、可靠、可查证的依据。

### 1.2 与通用百科的区别

这个网站虽然参考维基百科式协作，但不应做成普通百科页面堆积，而应突出四件事：

- 地方颗粒度更细，尽量到村
- 时间颗粒度更细，尽量到农历日期与节气节点
- 过程颗粒度更细，强调“具体怎么做”
- 证据颗粒度更细，强调“谁说的、从哪里来的、何时记录的”

## 2. 信息架构

### 2.1 顶层导航建议

- 首页
- 地区
- 节日
- 民俗条目
- 口述与文献
- 投稿
- 审核后台

### 2.2 主要对象关系

建议把网站核心拆成以下对象：

#### A. 地区 Region

表示行政区划或自然聚落。

核心字段建议：

- `id`
- `name`
- `standard_name`
- `type`：province / city / county / township / village
- `parent_id`
- `aliases`
- `ethnic_groups`
- `geo_lat`
- `geo_lng`
- `status`

说明：

- “行政村”和“自然村”最好分开支持，后面会很有用
- 同一个村落可能存在俗称、旧称、并村前名称，应保留别名

#### B. 节日或时间节点 Festival

表示农历节日、节气或固定时序民俗节点。

核心字段建议：

- `id`
- `name`
- `type`：festival / solar_term / seasonal_custom / life_ritual
- `lunar_month`
- `lunar_day`
- `is_leap_month_sensitive`
- `solar_term_name`
- `description`
- `status`

说明：

- 不要只支持“春节、中秋”，还要支持“腊月二十三祭灶”“二月二龙抬头”这类节点
- 后续可扩展到婚俗、丧俗、建房、祭祖等非节日类条目

#### C. 民俗条目 CustomEntry

这是网站最核心的数据对象，表示“某地区在某时间节点的某项民俗”。

核心字段建议：

- `id`
- `title`
- `region_id`
- `festival_id`
- `summary`
- `historical_origin`
- `symbolic_meaning`
- `ritual_process`
- `items_used`
- `food_offerings`
- `taboos`
- `oral_text`
- `participants`
- `inheritance_status`
- `current_change_notes`
- `content_status`

说明：

- 不要只存一大段正文，最好保留结构化字段
- 这样后面可以按“禁忌”“供品”“主持者”检索和比较

#### D. 来源证据 Source

用于支撑条目可信度。

核心字段建议：

- `id`
- `source_type`：oral / fieldwork / book / gazetteer / paper / photo / video / audio / web
- `title`
- `author_or_interviewee`
- `collector`
- `recorded_at`
- `location_text`
- `citation`
- `url`
- `license`
- `notes`

说明：

- “口述来源”和“文献来源”不要混在一起
- 最好给每个来源单独编号，方便条目引用

#### E. 条目来源关联 EntrySource

因为一个条目可能有多个来源，一个来源也可能支持多个条目。

核心字段建议：

- `id`
- `entry_id`
- `source_id`
- `supports_fields`
- `confidence_level`
- `editor_note`

#### F. 投稿 Submission

表示用户提交的新建或修改请求。

核心字段建议：

- `id`
- `submitter_user_id`
- `submission_type`：new_entry / edit_entry / add_source / fix_region / report_issue
- `target_entry_id`
- `payload_json`
- `statement_of_source`
- `contact_info`
- `status`
- `submitted_at`

#### G. 审核记录 Review

用于沉淀审核过程，保证可追溯。

核心字段建议：

- `id`
- `submission_id`
- `reviewer_user_id`
- `decision`：approved / rejected / needs_revision
- `review_comment`
- `reviewed_at`

#### H. 版本记录 Revision

用于保存公开条目的版本历史。

核心字段建议：

- `id`
- `entry_id`
- `version_number`
- `snapshot_json`
- `editor_user_id`
- `change_summary`
- `created_at`

## 3. 推荐的条目页面结构

一个高质量民俗条目页建议固定为以下模块：

1. 条目标题
2. 地区路径
3. 时间节点
4. 内容摘要
5. 仪式流程
6. 供品/器具/服饰/食物
7. 口述内容或祭词
8. 禁忌与注意事项
9. 象征意义与地方解释
10. 当代变化
11. 来源证据
12. 修订记录

这样做的好处是：

- 用户投稿更容易标准化
- 审核员更容易逐项核查
- 公开展示更容易比较不同地区差异

## 4. 投稿与审核流程

建议流程如下：

1. 用户选择地区
2. 用户选择节日或时间节点
3. 用户填写结构化民俗内容
4. 用户添加来源信息
5. 系统生成待审核稿
6. 审核员检查完整性、来源、措辞和重复情况
7. 审核员决定通过、退回修改或驳回
8. 通过后生成正式公开版本
9. 后续修改进入新一轮审核，并保留版本记录

## 5. 后台审核重点

审核员应重点看这些问题：

- 地区是否精确到正确村落
- 节日时间是否准确，是否注明农历
- 内容是在描述“本地真实做法”，还是泛泛介绍
- 来源是否具体可追踪
- 是否存在把传说当事实但没有标注的情况
- 是否存在地域混淆、年代混淆、民族混淆
- 是否包含对受访者不尊重或不准确的概括

## 6. 搜索与浏览设计

### 6.1 搜索维度

- 按地区搜索
- 按节日搜索
- 按农历日期搜索
- 按关键词搜索
- 按民俗类型搜索
- 按来源类型搜索

### 6.2 推荐的浏览入口

- 春节民俗专题
- 清明祭祖专题
- 某省民俗总览
- 某县村落民俗地图
- 同一节日在不同村落的差异对照

## 7. 内容规范建议

### 7.1 文风规范

- 用描述性语言代替判断性语言
- 区分“受访者说法”“地方传说”“研究者考证”
- 明确“仍在延续”“已弱化”“基本失传”“仅老人记得”等状态

### 7.2 可信度标记

建议给条目或字段增加可信度层级：

- A：多来源交叉验证
- B：单一高质量来源
- C：单人口述待核实
- D：线索性信息未核实

### 7.3 敏感内容处理

对于涉及家族内部资料、祖先牌位、祭词、面部影像、未公开族谱等内容，建议：

- 支持部分字段不公开
- 支持来源仅审核员可见
- 支持匿名化处理

## 9. 技术实现建议

如果后续你准备正式建站，建议优先满足这些能力：

- 支持结构化内容编辑
- 支持审核状态流转
- 支持全文搜索
- 支持版本历史
- 支持附件与媒体来源管理

数据库层面尤其重要的一点是：

“地区”“节日”“条目”“来源”“投稿”“审核”“版本”这几类表必须独立，不要从一开始就把所有信息塞进一张文章表。

## 10. 适合先做的开发顺序

### 第一阶段

- 建地区表
- 建节日表
- 建民俗条目表
- 建来源表
- 建投稿与审核表

### 第二阶段

- 做公开浏览页
- 做投稿表单
- 做审核后台
- 做版本对比

### 第三阶段

- 做高级搜索
- 做专题聚合页
- 做地区对比页
- 做地图或时间轴视图

## 11. 当前最值得先确认的两件事

### 11.1 内容最小标准

建议先定义“一个合格条目至少要包含什么”：

- 精确地区
- 节日或日期
- 具体做法
- 至少一条来源
- 是否仍在延续

### 11.2 技术栈

当前仓库还没有代码，所以正式开发前建议尽快定一个方向，例如：

- ASP.NET Core + PostgreSQL
- Next.js + PostgreSQL
- Django + PostgreSQL

不管选哪种，数据模型都可以沿用这份蓝图。
