# 命名与版本规范

## 文档信息
- 模块编号：`M00-02`
- 文件路径：`01_Standards/naming_and_versioning.md`
- 当前版本：`v1.2.0`
- 当前状态：`approved`

## 1. 适用范围
本规范适用于本项目中的以下对象：
- 仓库目录
- Markdown 文件
- 实体 ID
- 文档版本号
- 修改记录
- 多人或多 AI 协作时的文件占用与交接

本规范不定义民俗内容本体，不替代术语表、实体目录和字段表。

## 2. 文件夹命名规则
### 2.1 顶层目录
- 顶层目录必须使用两位数字前缀加英文名称，格式为：`NN_Name`。
- 两位数字只用于表达一级结构顺序，不用于表示版本号。
- 顶层目录一经被其他文件引用，不得随意改名。
- 新增顶层目录时，必须保证排序位唯一，不得跳号复用已有含义。

### 2.2 二级及以下目录
- 二级及以下目录名称必须使用小写英文或小写英文加下划线，禁止使用空格、中文、拼音缩写混写。
- 目录名称必须表达单一职责，不得在同层级出现同义目录，例如 `source` 与 `sources_archive` 同时表示来源主目录。
- 目录名称一律使用名词，不使用动词短语。

### 2.3 目录命名示例
- 合规：`00_Project_Brief`
- 合规：`03_Pilot_Data`
- 合规：`custom_instances`
- 不合规：`03-试点数据`
- 不合规：`Pilot Data`
- 不合规：`misc`

## 3. 文件命名规则
### 3.1 基本格式
- 文件名必须使用小写英文加下划线，扩展名统一为 `.md`。
- 文件名必须直接对应单一文档职责，不得用含糊名称，如 `notes.md`、`temp.md`、`final_new.md`。
- 稳定文档文件名中不得加入日期、作者名、版本号或“最终版”等字样。

### 3.2 类型前缀约定
- 模板文件统一使用前缀 `template_`，例如：`template_custom_instance.md`。
- 样板文件统一使用后缀 `_sample`，例如：`place_profile_sample.md`。
- 草案文件统一使用后缀 `_draft`，仅用于未进入正式规范层的预研文档，例如：`db_schema_draft.md`。
- 清单类文件统一使用后缀 `_checklist`，例如：`review_checklist.md`。

### 3.3 文件重命名规则
- 文件一旦在其他文档中被引用，重命名必须同步检查全部引用位置。
- 文件重命名前，必须确认是否影响模块队列、提示词库和交接文档中的路径。
- 文件重命名后，必须在该文件的修改记录中注明“原文件名 -> 新文件名”。

### 3.4 文件命名示例
- 合规：`field_dictionary_core.md`
- 合规：`template_source.md`
- 合规：`graph_schema_draft.md`
- 不合规：`字段表.md`
- 不合规：`template source.md`
- 不合规：`field_dictionary_core_v2.md`

## 4. 实体 ID 规则
### 4.1 通用格式
- 所有实体 ID 统一格式为：`前缀-四位顺序号`。
- 前缀必须全部使用大写英文字母。
- 顺序号必须从 `0001` 开始，按实体类型独立递增。
- ID 一经分配不得复用，即使对应记录被废弃或归档，也不得重新分配给其他对象。

### 4.2 首期核心实体前缀
| 实体类型 | 前缀 | 示例 |
|---|---|---|
| 地点 | `PLC` | `PLC-0001` |
| 节日 | `FES` | `FES-0001` |
| 仪式 | `RIT` | `RIT-0001` |
| 地方习俗实例 | `CUS` | `CUS-0001` |
| 来源资料 | `SRC` | `SRC-0001` |

说明：
- 首期核心实体当前统一为：地点、节日、仪式、地方习俗实例、来源资料。
- 首期核心实体是字段表、模板表和样板工程的优先建设对象。

### 4.3 扩展实体 / 预留前缀
| 实体类型 | 前缀 | 示例 |
|---|---|---|
| 媒体资料 | `MED` | `MED-0001` |
| 人群角色 | `ROL` | `ROL-0001` |
| 器物 | `OBJ` | `OBJ-0001` |
| 信仰观念 | `BEL` | `BEL-0001` |
| 时间规则 | `TIM` | `TIM-0001` |

说明：
- 扩展实体 / 预留前缀用于后续阶段扩展，不表示这些实体已在首期并行展开。
- `MED` 当前属于扩展 / 预留实体，不属于首期 5 个核心实体。
- 扩展 / 预留实体前缀不纳入首期常规发号，不作为首期字段表、关系目录和模板层的必做验收范围。
- 若后续模块需提前引用扩展实体，必须在文档中显式说明其为“扩展试用”或“预留前缀引用”。

### 4.4 分配规则
- 抽象词条与地方习俗实例必须使用不同前缀，不得混用。
- 同一实体只允许有一个主 ID；别名、俗称、旧称不得另建新 ID 替代主 ID。
- 新建记录时，先检查现有目录中是否已存在同一对象，确认不存在后再分配新 ID。
- 样板文档、模板文档中使用未知编号时，必须写为占位格式，如 `PLC-XXXX`，不得伪造真实编号。

### 4.5 引用规则
- 在跨文件引用实体时，优先使用“名称 + ID”的写法，避免只写名称导致歧义。
- 推荐写法：`某地村落（PLC-0012）`
- 不推荐写法：`某地村落`
- 当名称发生修订时，只允许更新名称，不允许改动已发布 ID。

## 5. 文档版本号规则
### 5.1 版本格式
- 所有正式规范文档使用版本格式：`v主版本.次版本.修订版本`。
- 示例：`v1.0.0`、`v1.2.3`

### 5.2 升级规则
- 主版本 `+1`：发生结构重构、章节职责改变、字段体系重排、下游必须同步调整时使用。
- 次版本 `+1`：新增完整章节、补充一组新规则、增加新的实体前缀或新增明确流程时使用。
- 修订版本 `+1`：修正文案、补示例、补充说明、修复歧义但不改变执行方式时使用。

### 5.3 进位规则
- 主版本升级时，次版本和修订版本必须归零。
- 次版本升级时，修订版本必须归零。
- 不允许跳级写版本号，例如从 `v0.1.0` 直接改为 `v2.0.0`，除非有明确结构性重构说明。

### 5.4 首期使用建议
- 当前仓库仍处于底座建设阶段，正式首版规范文档建议从 `v1.0.0` 起步。
- 若后续发生结构性重排，再按主版本规则升级。

## 6. 修改记录规则
### 6.1 每次修改必须记录的字段
- 日期：使用 `YYYY-MM-DD` 格式。
- 修改人：填写真实姓名、团队名或 AI 名称。
- 模块编号：例如 `M00-02`。
- 版本：填写修改后的文档版本号。
- 修改摘要：一句话说明本次改动内容。
- 影响范围：明确写出影响的下游模块或文件类型。

### 6.2 记录位置
- 每个正式规范文档末尾必须保留“修改记录”小节。
- 仓库级变更汇总由 `05_Operations/change_log.md` 统一记录。
- 若修改引发结构冲突或审核分歧，必须同步记录到 `05_Operations/review_log.md`。

## 7. 多人协作下的文件锁定与交接规则
### 7.1 单次会话范围
- 单次会话只处理一个模块，不同时处理多个模块文件。
- 一个模块若包含多个输出文件，必须在开始前明确本次涉及的文件路径。

### 7.2 文件锁定规则
- 同一时间只允许一个人或一个 AI 修改同一个目标文件。
- 开始编辑前，必须在任务说明或交接信息中明确写出“当前占用文件路径”。
- 若发现他人已声明占用同一文件，本轮不得并行改写该文件。

### 7.3 交接规则
- 交接时必须说明以下内容：
  - 当前模块编号
  - 已修改文件路径
  - 当前版本号
  - 已完成内容
  - 未决问题
  - 建议下一个动作
- 交接内容应优先写入 `05_Operations/handoff_template.md` 对应结构中，不得只留口头说明。

### 7.4 冲突处理规则
- 若出现两个版本同时修改同一文件，不得人工拼接合并后直接覆盖。
- 必须先在 `05_Operations/review_log.md` 记录冲突点，再由指定审核角色裁定保留方案。
- 裁定完成后，在 `05_Operations/change_log.md` 记录最终结果和影响范围。

## 8. 跨文件引用规则
- 引用其他文件时，必须使用仓库内相对稳定的文件路径或准确文件名，不得使用“见上文”“见另一个文件”等模糊写法。
- 引用实体时，优先写“名称 + ID”；引用规范时，优先写“文件名 + 章节名”。
- 当上游规范尚未定稿时，下游文档应写明“待以上游规范为准”，不得反向固化上游定义。
- 不允许在下游模板中私自新增上游尚未定义的核心字段名称；如确有需要，必须标记为“待字段表确认”。

## 9. 禁止事项
- 禁止同一含义使用多个文件名并行存在，例如 `glossary.md` 与 `term_dictionary.md` 同时承担术语表职责。
- 禁止随意修改已发布实体 ID。
- 禁止在不同文件中为同一实体创建多个主 ID。
- 禁止使用中文文件名、空格文件名或带版本号的稳定文件名。
- 禁止在未升级版本号的情况下大幅改动文档结构。
- 禁止在未完成引用同步检查的情况下重命名文件。
- 禁止未经审核或负责人确认，直接重构已被引用的文件路径或目录层级。
- 禁止跨模块扩写未分配任务的内容。
- 禁止用虚构编号、虚构来源或虚构地方事实填充样板文档。

## 10. 执行检查清单
- 新建目录前，是否确认目录层级与命名格式合规。
- 新建文件前，是否确认文件职责单一且命名符合约定。
- 新建实体前，是否确认该对象尚未拥有主 ID。
- 修改正式文档前，是否确认版本号升级方式正确。
- 提交结果前，是否补齐文档末尾修改记录。
- 交接前，是否写清已完成内容、未决问题和下一步建议。
